# Task1. Технологическая архитектура to-be

Файл `InsureTech-tech-architecture-to-be.drawio` содержит целевую технологическую архитектуру. Ниже коротко обоснование решений.

**Масштабирование.** Беру горизонтальное масштабирование stateless-сервисов через HPA. Рост нагрузки у InsureTech это прежде всего рост числа запросов, поэтому добавление экземпляров эффективнее наращивания ресурсов одного. Вертикальное масштабирование оставляю как точечный инструмент.

**Зоны доступности и фейловер.** Разворачиваю приложение по схеме Active-Active в нескольких зонах доступности одного региона Yandex Cloud. Балансировщик держу в отказоустойчивой конфигурации. Для цели 99,9 процента и RTO 45 минут этого достаточно, межрегиональное георезервирование избыточно и дорого.

**Время загрузки по регионам.** Требование одинакового времени загрузки для всех регионов закрываю через CDN перед веб-приложением. Статика и кеш отдаются у края сети, поэтому задержка слабо зависит от местоположения пользователя.

**Kubernetes.** Беру один управляемый региональный кластер с узлами в нескольких зонах, а не несколько независимых кластеров. При цели 99,9 процента в пределах одного региона это проще и надёжнее, чем федерация и синхронизация нескольких кластеров. Балансировка через L7 ALB, health check на балансировщике и liveness и readiness пробы на подах.

**База данных.** Беру управляемый PostgreSQL в HA-конфигурации. Мастер и синхронная реплика в разных зонах, автоматический фейловер на базе Patroni. RPO 15 минут закрываю синхронной репликой и непрерывным WAL, при этом фактический RPO близок к нулю. RTO 45 минут закрываю автоматическим переключением на реплику. Резервные копии и WAL храню в Object Storage для восстановления на момент времени.

**Шардирование.** Не применяю. Общий объём данных 50 GB помещается на одном primary, а шардирование оправдано при кратно больших объёмах и нагрузке на запись. Для масштабирования чтения добавляю read-реплики.
