# Task4. Проектирование продажи ОСАГО

Файл `InsureTech-C4-osago.drawio` дополняет диаграмму из Task3 продажей ОСАГО. Ниже решения по вопросам задания.

**Хранилище osago-aggregator.** Сервису нужно своё хранилище. Опрос страховых идёт асинхронно до 60 секунд, а сервис работает в нескольких экземплярах, поэтому состояние заявок и пришедших предложений нельзя держать в памяти одного процесса. Беру Redis, он даёт быстрый доступ и механизм pub/sub.

**API osago-aggregator для core-app.** core-app отправляет команду создать заявку и получает идентификатор заявки. Дальше osago-aggregator асинхронно опрашивает страховые и по мере получения публикует каждое предложение. Это комбинирование синхронного и асинхронного взаимодействия без вложения одного в другое.

**Интеграция core-app и osago-aggregator.** Команду создания заявки делаю синхронной, на ней применяю Timeout и Retry. Предложения возвращаются асинхронно через Redis pub/sub по идентификатору заявки.

**API веб-приложения и core-app.** Беру Server push через WebSocket, потому что предложения приходят по мере готовности, а при 2500 одновременных пользователях постоянный опрос неэффективен. На схеме это отдельная стрелка, отличная от REST.

**Несколько экземпляров.** Предложение может прийти в любой экземпляр core-app, а WebSocket клиента держит конкретный экземпляр. Маршрутизацию между экземплярами делаю через Redis pub/sub по идентификатору заявки.

**Паттерны отказоустойчивости.** На вызовах страховых применяю Timeout 60 секунд, Retry на временные ошибки и Circuit Breaker на каждую компанию. На входе API партнёров применяю Rate Limiting.
